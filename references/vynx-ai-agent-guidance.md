# VYNX AI-Agent Usage Guidance

Use this file to tell agents how to work with the methodology as an operating system rather than a generic knowledge base.

## Read Order

Start with:

1. `SKILL.md`
2. `references/vynx-methodology-summary.md`
3. `references/vynx-principles.md`
4. `references/core-principles.md`
5. `references/domain-industrial-engineering-manufacturing.md`
6. `references/diagnostics.md`

Then choose the right framework:

- use `diagnostics.md` for stalled deals, proposal conversion issues, distributor inconsistency, or RFQ problems
- use `operating-model.md` and `stakeholders.md` for ownership, governance, handoffs, and cadence design
- use `measurement.md` and `vynx-executive-dashboard.md` for KPI packs, dashboards, scorecards, and metric logic
- use `maturity.md` and `vynx-maturity-assessment.md` for phased improvement, sequencing, and readiness assessment
- use `manufacturing-use-cases.md` and `vynx-fillable-templates.md` when shaping outputs for industrial field reality
- use `vynx-consulting-workstreams.md`, `vynx-client-deliverable-stack.md`, and `vynx-productized-offer.md` for consulting packaging or client-facing deliverables

## Generate Outputs Such As

- root-cause diagnosis
- stage-gate architecture
- decision rules
- review cadence
- fillable templates
- KPI frameworks
- distributor tiering models
- rollout plans
- workshop agendas
- executive summaries

## Always Assume

- long cycles
- multiple stakeholders
- plant visits and technical validation matter
- service credibility affects conversion
- distributors vary widely in capability

## Avoid

- generic sales-enablement language
- SaaS examples
- tool-first recommendations
- training-only answers
- KPI lists with no decision logic
- proposals that ignore implementation and commissioning risk

## Operating Standard

Favor outputs that a client can use tomorrow, not conceptual summaries. If context is incomplete, state assumptions and keep the design modular.
